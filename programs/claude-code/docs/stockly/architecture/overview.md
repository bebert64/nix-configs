# Architecture Overview

Stockly is a B2B e-commerce platform connecting **retailers** (demanders) with **suppliers**. A retailer lists products on their storefront; when a consumer buys, Stockly orchestrates purchasing from the supplier, shipping, invoicing, and after-sale.

## Tech Stack

| Layer | Technology |
|-------|-----------|
| Backend | Rust (2024 edition) |
| Frontend | Angular + TypeScript |
| Database | PostgreSQL via Diesel ORM |
| RPC | gRPC (grpcio) with proto3 |
| Package managers | `cargo` (Rust), `pnpm` (JS/TS) |
| Environment | direnv + Nix |
| CI/CD | Jenkins + Bors merge queue |
| Monitoring | Sentry, OpenMetrics/Prometheus |
| Object storage | S3 (legacy helper) |
| Email | Sweego SDK |

## Service Internal Structure (convention)

Each service follows a consistent layout:

```
<service>/
├── Core/                # Shared types, IDs, lightweight structs
│   └── src/
├── Schema/              # Diesel schema + migrations
│   ├── src/schema.rs    # Auto-generated by Diesel CLI
│   └── migrations/      # Timestamped SQL migrations (up.sql only)
├── Service/             # Main binary + business logic
│   ├── src/
│   │   ├── main.rs      # Entry point (Sentry init, metrics, gRPC server)
│   │   ├── lib.rs       # Module declarations, prelude, server builder
│   │   ├── grpc/        # RPC handlers + proto definitions
│   │   └── <domain>/    # Business logic modules
│   └── post_deployment_tests/  # Post-deploy smoke tests
└── MismatchDetection/   # (operations-specific) data consistency checks
```

Not all services have all parts — smaller services may combine Schema and Service.

## Communication

Services communicate primarily via **gRPC**. Proto files live next to the service that defines them (e.g., `operations/Service/src/grpc/demand/demand.proto`). Shared protos live in `protos/` at the repo root.

Cross-service dependencies are explicit in `Cargo.toml` via proto crate imports (e.g., `stocks_proto_items.workspace = true`).

## Key Shared Libraries (`lib/rust/`)

| Crate | Purpose |
|-------|---------|
| `InternalError` | Error type (`InternalError`/`InternalResult`), `try_or_wrap!`, Sentry integration |
| `grpc_helpers/` | Client, Server, Codegen, HttpToGrpc — full gRPC stack |
| `DieselHelpers` | Transaction helpers, advisory locks, JSON helpers, filters |
| `model_id` | Typed ID wrappers (`OrderLineId`, `PurchaseId`, etc.) |
| `validation` | Field validation, `DynFieldInvalidity`, `PreValidated<T>` |
| `chrono_helpers` | Date/time utilities, `jour_franc` (business days) |
| `sort_helpers` | `Sorted` trait for immutable sorting |
| `internal_error` | Re-exports, `err_ctx!`, `err_msg` |
| `countries` | `Language`, `Alpha3`, `Currency` typed wrappers |
| `price` | Price arithmetic with currency awareness |
| `float_ext` | Tolerant float comparisons (`FloatExt`) |
| `iter_ext` | Iterator extensions for `InternalResult` |
| `serde_helpers` | Serde utilities, `assert_json_eq` test helper |
| `axum_helpers` | Axum HTTP server wrapper |
